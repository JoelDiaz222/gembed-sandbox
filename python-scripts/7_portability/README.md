# Benchmark 7 — Portability

## Purpose

Compares embedding-generation throughput across three database adapters that share
the **same Gembed Rust core and the same EmbedAnything backend**, but expose it
through different DBMSs:

| Adapter          | System     | SQL / Command                                                  |
|------------------|------------|----------------------------------------------------------------|
| **pg_gembed**    | PostgreSQL | `SELECT embed_texts('embed_anything', model, ARRAY[...])`      |
| **mysql_gembed** | MySQL      | `SELECT EMBED_TEXTS('embed_anything', model, JSON_ARRAY(...))` |
| **redis_gembed** | Redis      | `G.EMBEDS embed_anything model text [text …]`                  |

The goal is to demonstrate that the **Gembed Rust core is portable** across different
database engines with consistent throughput, while attributing any differences to
the adapter overhead (SQL planning, connection protocol, etc.).

## Models Tested

| Model                                    | Dim  | Size class   |
|------------------------------------------|------|--------------|
| `sentence-transformers/all-MiniLM-L6-v2` | 384  | Small / fast |
| `BAAI/bge-large-en-v1.5`                 | 1024 | Large        |

The two models span a range of output dimensionality and model size, showing
how adapter overhead compares to raw inference cost.

## What Is Measured

For a fixed input batch of `--sizes` texts:

1. **Wall-clock time** — total time for the database to embed all texts (including
   serialization and protocol overhead).
2. **Throughput** — texts per second.
3. **Resource usage** — Python process CPU/memory; DB-side process CPU/memory
   (PostgreSQL backend PID, MySQL connection OS thread PID, Redis server PID).

> **Note**: a warm-up call (not timed) is performed for each adapter × model
> pair before measurement to ensure model weights are loaded into the in-process
> cache.

## Prerequisites

### PostgreSQL (pg_gembed)

- PostgreSQL 17+ with both `vector` and `pg_gembed` extensions installed.
- Connection config: `localhost:5432`, database and user defined in `utils/benchmark_utils.py`.

### MySQL (mysql_gembed)

- MySQL with the `component_mysql_gembed` component installed.
- Server running at `localhost:3306`, accessible as `root` (no password by default in dev).
- `mysql-connector-python` installed in the venv:
  ```bash
  pip install mysql-connector-python
  ```

### Redis (redis_gembed)

- Redis 7+ running at `localhost:6379` with `redis_gembed.so` loaded:
  ```bash
  redis-server --loadmodule /path/to/redis_gembed.so
  ```
- `redis` Python client installed in the venv:
  ```bash
  pip install redis
  ```

## How to Run

### Single run

```bash
# From the python-scripts directory (with venv active)
python 7_portability/benchmark.py --sizes 1024 --run-id test01
```

Optional: run only a subset of adapters:

```bash
python 7_portability/benchmark.py --sizes 1024 --adapters pg redis
```

### Via orchestrator

The benchmark is registered in `benchmark_config.yaml` as `7_portability` and
runs with a fixed size of `1024` texts (15 runs by default). The orchestrator
handles concatenation of run CSVs and plot/table generation.

```bash
./orchestrator.sh 7_portability
```

## Output

- `output/benchmark_<run_id>_run.csv` — raw per-run CSV
- After orchestration: throughput bar chart, resource charts (PDF + PNG), and
  LaTeX tables generated by `plot_utils.generate_plots_b7()` and `table_utils`.
